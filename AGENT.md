# AGENTS.md

## 프로젝트 원칙

이 프로젝트는 데모를 만들기 위한 저장소가 아니다.

모든 구현은 특정 데모 시나리오가 아니라 일반적인 사용자 요청에서도 동작할 수 있는 구조를 기준으로 한다.

데모는 구현 결과를 보여주는 수단일 뿐이며, 구현 방향을 결정하는 기준이 되어서는 안 된다.

---

## 구현 규칙

* 특정 기관, 특정 공고, 특정 결과를 위한 하드코딩을 피한다.
* 데모 결과를 좋게 보이게 하려고 임시 데이터나 임시 로직을 넣지 않는다.
* 새 기능은 일반적인 입력에서도 동작해야 한다.
* 외부 데이터가 없거나 결과가 비어도 안전하게 실패해야 한다.
* 근거가 부족한 분석은 단정하지 않고 `warnings`, `verification_notes`, `needs_verification` 등으로 표시한다.
* 코드는 가능한 간결하게 작성한다.
* 불필요한 추상화, 과도한 클래스 분리, 사용하지 않는 옵션 추가를 피한다.
* 하나의 PR은 하나의 목적만 가진다.

---

## 승인 없이 하지 말 것

다음 작업은 구현 전에 반드시 사용자 또는 저장소 소유자의 승인을 받는다.

* 새로운 MCP tool 추가
* 새로운 외부 데이터 소스 연동
* 새로운 dependency 추가
* 특정 기관 코드나 seed 데이터 추가
* 기존 tool의 입력/출력 스키마 변경
* README에서 프로젝트 범위를 넓혀 보이게 하는 문구 추가
* 데모용 문서나 데모용 출력 파일 추가

승인을 받기 전에는 설계 제안까지만 작성하고 구현하지 않는다.

## 테스트 및 검증 규칙

기능을 추가하거나 수정할 때는 자동 테스트뿐 아니라 실제 실행 결과도 확인한다.

PR에는 가능한 한 다음 검증 결과를 포함한다.

* 관련 단위 테스트 실행 결과
* 전체 테스트 실행 결과
* 린트 실행 결과
* 실제 MCP 서버 또는 CLI를 통해 도구를 호출한 결과
* 실패 케이스 또는 0건 결과가 어떻게 반환되는지 확인한 결과

예시는 다음과 같다.

```bash id="ru9519"
python -m pytest tests/test_public_jobs_tool.py -q
python -m pytest -q
python -m ruff check .

python -m kr_gov_job_mcp.server --list-tools
python -m kr_gov_job_mcp.server --call-tool search_public_jobs --input '{"keyword":"정보통신","limit":2,"ongoing_only":false}'
python -m kr_gov_job_mcp.server --call-tool fetch_job_detail --input '{"job_id":"302324"}'
```

검증 결과는 단순히 “테스트 통과”라고 쓰지 말고, 실제 명령어와 핵심 출력 요약을 PR에 남긴다.

외부 API를 호출하는 기능은 네트워크 상태나 원본 데이터 변경으로 결과가 달라질 수 있으므로, PR에는 실행 날짜와 확인한 주요 응답 필드를 함께 기록한다.

## PR 규칙

모든 PR 설명과 리뷰 코멘트는 한글로 작성한다.

PR 설명에는 최소한 다음 항목을 포함한다.

```md id="tuciqk"
## 변경 내용
-

## 왜 필요한가
-

## 검증
-

## 실제 호출 결과
-

## 주의할 점
-
```

리뷰에서는 다음을 확인한다.

* 데모용 하드코딩이 있는가?
* 일반적인 사용 상황에서도 동작하는가?
* 실패 케이스가 처리되어 있는가?
* 테스트가 있는가?
* 실제 서버 또는 CLI 호출 결과가 있는가?
* 코드가 불필요하게 복잡하지 않은가?
* 문서가 실제 구현보다 과장되어 있지 않은가?

## 현재 MVP 범위

현재 MVP는 다음 기능에 집중한다.

* Job-ALIO 채용공고 검색
* 지역명 기반 근무지역 코드 조회
* 공고 상세 조회
* 첨부파일 및 직무기술서 후보 식별
* NCS 분류 후보 정리
* 근거 링크와 확인 필요 사항 반환

다음 기능은 후속 단계로 분리한다.

* ALIO/클린아이 자동 수집
* 전체 기관 코드 자동 수집
* PDF/HWP/HWPX 자동 본문 추출
* 자연어 질의 전체 orchestration
* 기관 전략/약점 분석 고도화
* 배포/HTTP/CORS 고도화
* 지원자 준비 리포트 생성

## 작업 전 체크리스트

작업을 시작하기 전에 다음을 확인한다.

* 이 작업이 현재 MVP 범위에 포함되는가?
* 특정 데모만을 위한 구현은 아닌가?
* 기존 도구를 확장하면 되는가, 새 도구가 꼭 필요한가?
* 승인 없이 진행해도 되는 작업인가?
* 실패했을 때 사용자에게 설명 가능한가?
* 테스트를 추가할 수 있는가?
* 실제 서버 또는 CLI 호출로 결과를 확인할 수 있는가?
